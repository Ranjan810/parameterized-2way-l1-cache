# Evaluation Report

This document summarizes the functional validation of the SystemVerilog cache controller against an independent Python architectural golden model. The RTL was evaluated using identical memory traces and cache configurations, and the architectural statistics generated by both implementations were compared.

Validation was performed for:

- 4 memory access workloads
- 2 replacement policies (LRU and FIFO)
- Prefetcher Enabled and Disabled
- 16 total validation configurations

For every configuration, the following metrics were verified:

- Cache Hits
- Cache Misses
- Hit Rate
- Dirty Evictions (Write-Backs)
- Memory Reads
- Memory Writes
- Total Memory Traffic
- Average Memory Access Time (AMAT)

Across all evaluated configurations, the RTL produced identical architectural results to the Python golden model.

---

# Validation Workloads

The following memory access patterns were used to evaluate the cache controller.

| Workload | Description |
|----------|-------------|
| Sequential | Contiguous memory accesses with high spatial locality |
| Random | Uniformly distributed random memory accesses |
| Stride | Fixed-stride memory accesses designed to stress cache mapping |
| Thrashing | Alternating accesses producing frequent conflict misses |

Each workload contains **1000 memory accesses** for RTL validation. Additional Python simulations were also performed with **5000 accesses** to verify scalability.

# Average Memory Access Time (AMAT)

## LRU Replacement Policy

| Workload | Prefetch OFF | Prefetch ON |
|----------|-------------:|------------:|
| Sequential | 3.25 | 2.00 |
| Random | 6.96 | 6.97 |
| Stride | 7.00 | 7.00 |
| Thrashing | 4.96 | 4.96 |

---

## FIFO Replacement Policy

| Workload | Prefetch OFF | Prefetch ON |
|----------|-------------:|------------:|
| Sequential | 3.25 | 2.00 |
| Random | 6.96 | 6.97 |
| Stride | 7.00 | 7.00 |
| Thrashing | 4.93 | 4.93 |

The next-line prefetcher provides the greatest performance improvement for **sequential workloads**, reducing AMAT from **3.25 cycles to 2.00 cycles** by fetching the next cache line before it is requested. For random and stride workloads, prefetching generates additional memory traffic without improving locality, resulting in little or no AMAT improvement. Thrashing workloads show negligible change because conflict misses dominate cache behavior. 

---

# Memory Traffic Analysis

The prefetcher trades increased memory bandwidth for reduced processor stall time.

| Workload | Effect of Prefetching |
|----------|-----------------------|
| Sequential | Slight increase in memory traffic with a significant increase in hit rate and reduction in AMAT |
| Random | Approximately doubles memory reads due to speculative accesses with almost no performance improvement |
| Stride | Doubles memory reads while providing no additional cache hits |
| Thrashing | Increases memory reads but conflict misses dominate overall performance |

These results illustrate that a simple next-line prefetcher is highly effective only when strong spatial locality exists.

# Architectural Validation Summary

The RTL implementation was validated against the Python architectural model using all combinations of:

- 4 workloads
- 2 replacement policies
- Prefetch Enabled / Disabled

A total of **16 architectural validation configurations** were evaluated.

For every configuration, the following metrics matched exactly:

- Cache Hits
- Cache Misses
- Hit Rate
- Dirty Evictions (Write-Backs)
- Memory Reads
- Memory Writes
- Total Memory Traffic
- Average Memory Access Time (AMAT)

This demonstrates that the SystemVerilog RTL faithfully reproduces the behavior of the independent Python architectural model across diverse access patterns and cache configurations.

---

# Scalability Verification

In addition to the RTL simulations performed with **1000 memory accesses**, the Python reference model was also evaluated using **5000-access** traces. The larger simulations preserve the same architectural trends observed in the RTL results, demonstrating that the cache behavior scales consistently with increasing workload size while maintaining the expected characteristics of each replacement policy and prefetch configuration.
